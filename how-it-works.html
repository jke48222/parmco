§ HOW IT WORKS

From phone tap to motor spin.

Every layer, in order — UI event, radio frame, D-Bus call, C dispatch, PWM duty, register write.

§ 01 / Handshake

Establishing the BLE link.

Sequence

On launch, the PARMCO iOS app initialises a CoreBluetooth CBCentralManager and scans for peripherals advertising the PARMCO service UUID. The Pi runs ble_server as a systemd unit that registers a custom GATT application with BlueZ over D-Bus and advertises as PARMCO-RP4.

When the phone discovers the peripheral it connects, discovers two characteristics (command, write; telemetry, notify), and subscribes so the live RPM display can update.

A live handshake, once per connection. Steady-state is step ∞, the telemetry stream.

Pairing

iOS triggers a BLE pairing exchange on first connection to a peripheral with writable characteristics. Without a registered agent on the Pi, BlueZ has no handler and the connection hangs indefinitely. ble_server registers a NoInputNoOutput agent at startup that auto-accepts using Just Works — no PIN, no passkey, no user interaction.

After iOS forgets the device, the agent accepts re-pairing on the next connection without any console input on the Pi. This behavior is verified explicitly in the CP3 demo script.

§ 02 / Protocol

Commands, in plain ASCII.

Commands

Commands travel as plain UTF-8 strings written to the command characteristic (…DEF1). Write-without-response is used alongside standard write — this avoids triggering an unnecessary SMP authentication exchange on iOS and reduces round-trip latency for speed updates.

CommandEffect
FWDSet direction to forward (clockwise)
REVSet direction to reverse (counterclockwise)
STARTEnable the motor at current speed setting
STOPDisable the motor; coasts to a stop
SPD:XXSet speed, percent 0 to 100 — e.g. SPD:75
RPM:<value>Set target RPM for Maintain mode
AUTO:MAINTAINSwitch to closed-loop RPM maintenance
AUTO:MATCHSwitch to reference-tracking mode (GPIO23)
Dispatch

On the Pi, parse_command() receives each UTF-8 byte array from BlueZ, null-terminates it, and dispatches on exact string matches. Unknown commands are logged and silently ignored.

ble_server.c · parse_command dispatch 
static void parse_command(const char *cmd)
{
    if (strcmp(cmd, "FWD") == 0) {
        strncpy(motor_state.direction, "FWD", sizeof(motor_state.direction));

    } else if (strncmp(cmd, "SPD:", 4) == 0) {
        int spd = atoi(cmd + 4);
        /* clamp to [0, 100] */
        motor_state.speed = (spd < 0) ? 0 : (spd > 100) ? 100 : spd;

    } else if (strcmp(cmd, "START") == 0) {
        motor_state.running = 1;
    } else if (strcmp(cmd, "STOP") == 0) {
        motor_state.running = 0;
    }
}
§ 03 / Telemetry

RPM back, in real time.

Channel

The telemetry characteristic (…DEF2) carries RPM readings back to the phone as ASCII strings in the form RPM:<int> — e.g. RPM:4523. BlueZ pushes updates to the subscribed phone via PropertiesChanged signals on the characteristic's D-Bus object path.

Cadence
StateCadenceRationale
Motor running5 Hz · 200 msResponsive display; matches the 250 ms RPM sample window.
Motor idle1 Hz · 1 sKeeps the phone's live indicator honest without spamming BLE.
Computation

RPM is measured from an IR sensor on the propeller, wired to GPIO17. pigpio's alert callback counts rising edges with an 800 µs debounce filter. With three blades (three pulses per revolution), the firmware computes:

freq_hz = (pulse_count × 1000) / elapsed_ms
rpm     = (freq_hz × 60) / pulses_per_rev  /* pulses_per_rev = 3 */
§ 04 / Drive

Motor control.

Wiring

The L293D H-bridge is wired to three GPIO pins. Direction is set by two logic outputs; speed by hardware PWM on the enable pin.

GPIOL293D pinFunction
GPIO18Pin 1 · EN1,220 kHz DMA PWM · speed 0–100%
GPIO20Pin 7 · IN2Direction bit B · LOW for forward
GPIO21Pin 2 · IN1Direction bit A · HIGH for forward
GPIO17IR sensorActual RPM feedback
GPIO23Reference inputExternal RPM reference · Match mode
Modes

Manual

The phone sends SPD:XX. The firmware converts the percentage to a PWM duty cycle directly (duty = speed_percent × 255 / 100). No feedback loop.

Maintain

The phone sends a target RPM via RPM:<value>. A proportional controller (Kp = 0.04) runs every 250 ms, nudging PWM duty up or down against the IR-measured actual RPM. A ±30 RPM deadband prevents hunting at steady state.

Match

The motor tracks an external reference signal on GPIO23. The same proportional controller runs on the error between reference and actual RPM.

motor_control_cp3.c · maintain controller 
static void update_maintain_mode(void)
{
    float error  = (float)target_rpm - actual_rpm;
    int   adjust = (int)(KP * error);   /* KP = 0.04f */

    /* ±30 RPM deadband, suppress hunting at steady state */
    if (error > -30.0f && error < 30.0f) adjust = 0;

    set_pwm_raw(pwm_duty + adjust);
}
Assembly

Direct GPIO writes

Direction changes call gpio_write(), an ARMv7 assembly routine that writes directly to the BCM2711 GPIO SET and CLR registers through a memory-mapped /dev/gpiomem region. Writing a bitmask to GPSET0 (offset 0x1C) drives a pin HIGH; writing to GPCLR0 (offset 0x28) drives it LOW. These are atomic — no read-modify-write cycle needed.

gpio_asm.s · gpio_write (AArch32) 
gpio_write:                         @ (pin: r0, value: r1)
    ldr     r2, =gpio_base_addr
    ldr     r2, [r2]              @ load mmap'd base pointer
    mov     r3, #1
    lsl     r3, r3, r0             @ bitmask = 1 << pin
    cmp     r1, #0
    beq     .Lclear_pin
    str     r3, [r2, #0x1C]       @ GPSET0 → pin HIGH
    bx      lr
.Lclear_pin:
    str     r3, [r2, #0x28]       @ GPCLR0 → pin LOW
    bx      lr
§ 05 / Reliability

Boot & recovery.

systemd

ble_server runs as a systemd service (parmco-ble.service) that starts automatically after bluetooth.service on every boot. If the process crashes, systemd restarts it after 2 seconds. On restart, the GATT application and pairing agent are re-registered from scratch.

CP3's recovery test goes like this: forget the device on the phone, pull the Pi's power, plug it back in. Within about 45 seconds the phone finds it again on its own. No keyboard, no app restart.

parmco-ble.service 
[Unit]
Description=PARMCO BLE GATT Server
After=bluetooth.service
Requires=bluetooth.service

[Service]
ExecStart=/usr/local/bin/ble_server
Restart=on-failure
RestartSec=2
Type=simple

[Install]
WantedBy=multi-user.target
